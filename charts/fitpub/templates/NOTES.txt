Thank you for installing {{ .Chart.Name }}.

Your release is named {{ .Release.Name }}.

Release status:

  helm status {{ .Release.Name }} -n {{ .Release.Namespace }}
  kubectl rollout status deployment/{{ include "fitpub.fullname" . }} -n {{ .Release.Namespace }}
  kubectl get pods -l app.kubernetes.io/instance={{ .Release.Name }} -n {{ .Release.Namespace }}

{{- if .Values.ingress.enabled }}

You can access the application at:
{{- range .Values.ingress.hosts }}
  http{{ if $.Values.ingress.tls }}s{{ end }}://{{ .host }}{{ (index .paths 0).path }}
{{- end }}

{{- else if contains "NodePort" .Values.service.type }}

Get the application URL by running:
  export NODE_PORT=$(kubectl get svc {{ include "fitpub.fullname" . }} -o jsonpath="{.spec.ports[0].nodePort}")
  export NODE_IP=$(kubectl get nodes -o jsonpath="{.items[0].status.addresses[0].address}")
  echo http://$NODE_IP:$NODE_PORT

{{- else if contains "LoadBalancer" .Values.service.type }}

It may take a few minutes for the LoadBalancer IP to be available.
You can watch the status by running:
  kubectl get svc -w {{ include "fitpub.fullname" . }}

{{- else if contains "ClusterIP" .Values.service.type }}

The application is exposed via ClusterIP.
You can port-forward to access it locally:

  kubectl port-forward svc/{{ include "fitpub.fullname" . }} 8080:{{ .Values.service.port }} -n {{ .Release.Namespace }}

Then open http://127.0.0.1:8080 in your browser.

{{- end }}

Important:
- Non-secret environment variables are rendered from `config` into a ConfigMap
- Secret environment variables come from `applicationSecret.data` or `applicationSecret.existingSecret`
- FitPub requires an external PostgreSQL database with PostGIS enabled
- FITPUB_BASE_URL must be the canonical public URL without a trailing slash
- A PersistentVolumeClaim is created for `{{ .Values.persistence.mountPath }}` by default
- ConfigMap: {{ include "fitpub.configMapName" . }}
- Secret: {{ include "fitpub.secretName" . }}
{{- if not .Values.productionChecks.enabled }}
- Consider enabling `productionChecks.enabled=true` in production values to fail fast on missing or inconsistent settings
{{- end }}
{{- if gt (int .Values.replicaCount) 1 }}
- Multiple replicas should be used only after validating FitPub background jobs, federation processing and shared storage behavior
{{- end }}

Useful checks:

  kubectl logs deployment/{{ include "fitpub.fullname" . }} -n {{ .Release.Namespace }}
  kubectl exec deployment/{{ include "fitpub.fullname" . }} -n {{ .Release.Namespace }} -- sh -c 'touch {{ .Values.persistence.mountPath }}/.write-test && rm {{ .Values.persistence.mountPath }}/.write-test'

Health endpoint from inside the cluster:

  kubectl run fitpub-healthcheck --image=curlimages/curl --restart=Never --rm -i -n {{ .Release.Namespace }} --command -- curl -fsS http://{{ include "fitpub.fullname" . }}:{{ .Values.service.port }}/actuator/health

If the pod does not become Ready, the cause is almost always the database:
either it is unreachable, or it is plain PostgreSQL without the PostGIS extension
(the V1__enable_postgis migration then fails). Check with:

  kubectl logs deployment/{{ include "fitpub.fullname" . }} -n {{ .Release.Namespace }}

Debugging:
If the pod fails to start, enable diagnostic mode to exec into the container
without startup or probe failures blocking you:

  helm upgrade {{ .Release.Name }} fitpub/fitpub \
    --reuse-values \
    --set diagnosticMode.enabled=true

  kubectl exec -it deployment/{{ include "fitpub.fullname" . }} -n {{ .Release.Namespace }} -- sh

Remember to disable diagnostic mode afterwards:

  helm upgrade {{ .Release.Name }} fitpub/fitpub \
    --reuse-values \
    --set diagnosticMode.enabled=false
